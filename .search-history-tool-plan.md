# LLM Tool: `search_history` — Fuse.js Fuzzy Search for the LLM

**Status:** Planned — not implementing today

## Goal

Give the LLM a tool to fuzzy-search the user's **entire** conversation history (topics and messages across all conversations) using the same fuse.js pipeline built into `GlobalSearch.tsx`. The LLM can call this to find relevant context from past conversations without needing the user to manually search.

> **Note:** Unlike `read_messages` and `list_messages` (which are scoped to the current topic), this tool searches across ALL topics. This is an intentional scope expansion and requires explicit user consent via a dedicated toggle.

## Background

The app currently has 4 LLM tools defined in `src/services/llmService.ts:69-145`:

- `update_scratchpad` — persist memory across turns
- `read_messages` — fetch specific messages by ID
- `list_messages` — directory of all messages (IDs + 150-char snippets, no search)
- `ask_user` — ask the user a follow-up question

These are:

- **Assembled** in `src/store/ChatStore.ts:989-999`
- **Executed** via `onExecuteTool` callback in `src/store/ChatStore.ts:878-987`
- **Gated** by auth store toggles (`messageRetrievalEnabled`, `askUserEnabled`)
- **Orchestrated** by the tool loop in `src/services/llmService.ts:1216-1377` (max 10 iterations)

The proposed `search_history` tool fills the gap between `list_messages` (firehose of everything) and `read_messages` (exact ID lookup) — providing targeted, relevance-ranked search.

```ts
    parameters: {
      type: 'object',
      properties: {

          type: 'string',
          description: 'The search query — can be multiple words, handles typos',

          type: 'string',
          enum: ['topics', 'messages', 'both'],
          description: 'What to search: topic names, message contents, or both. Default: both',

    },
  },

```

## Gating

Add a user toggle in the **Settings page** (where `messageRetrievalEnabled` and `askUserEnabled` toggles live), NOT in ProviderCard (which holds per-provider capability flags like `supportsWebSearch`).

**Option A:** Reuse `messageRetrievalEnabled` — simplest, no new UI needed. If user allows message retrieval, search is included.

**Option B:** New toggle `historySearchEnabled` in auth store — more granular control.

Recommendation: **Option B** for explicit user consent. The toggle label should clarify the cross-topic scope (e.g., "Allow LLM to search across all conversations").

> **Dependency:** `historySearchEnabled` is nested inside the `messageRetrievalEnabled` check during tool assembly. If message retrieval is disabled, history search is also disabled regardless of its own toggle state. This is intentional but should be surfaced in the UI (e.g., greyed out when message retrieval is off).

## Execution

In `src/store/ChatStore.ts`, within the `onExecuteTool` callback (around line 878), add a handler:

```ts
if (toolName === 'search_history') {
  try {
    const parsedArgs = JSON.parse(argsJson) as { query?: string; mode?: string };
    const query = parsedArgs.query ?? '';
    const mode = parsedArgs.mode ?? 'both';

    if (!query.trim()) {
      return 'Error: query parameter is required and must not be empty.';
    }

    const results: string[] = [];

    if (mode === 'both' || mode === 'topics') {
      const topics = await athenaDb.topics
        .toCollection()
        .filter((t) => !t.isDeleted)
        .toArray();
      const fuse = new Fuse(topics, {
        keys: ['name'],
        threshold: 0.4,
        includeScore: true,
        minMatchCharLength: 2,
      });
      const topicResults = fuse.search(query).slice(0, 5);
      if (topicResults.length > 0) {
        results.push('## Topic matches');
        topicResults.forEach((r) => {
          results.push(`- "${r.item.name}" (topic ID: ${r.item.id})`);
        });
      }
    }

    if (mode === 'both' || mode === 'messages') {
      const messages = await athenaDb.messages
        .toCollection()
        .filter((m) => !m.isDeleted)
        .toArray();
      const fuse = new Fuse(messages, {
        keys: ['content'],
        threshold: 0.4,
        includeScore: true,
        includeMatches: true,
        minMatchCharLength: 2,
        ignoreLocation: true,
      });
      const messageResults = fuse.search(query).slice(0, 5);
      if (messageResults.length > 0) {
        // Fetch parent topic names for context
        const topicIds = Array.from(new Set(messageResults.map((r) => r.item.topicId)));
        const topics = await athenaDb.topics.bulkGet(topicIds);
        const topicNames = new Map(topics.filter(Boolean).map((t) => [t!.id, t!.name]));

        results.push('## Message matches');
        messageResults.forEach((r) => {
          const topicName = topicNames.get(r.item.topicId) || 'unknown topic';
          const snippet = r.item.content.substring(0, 300).replace(/\n/g, ' ');
          results.push(`- [ID: ${r.item.id.slice(0, 8)}] (${topicName}): "${snippet}${r.item.content.length > 300 ? '...' : ''}"`);
        });
        results.push('\nUse read_messages with any of these IDs to see full content.');
      }
    }

    return results.length > 0 ? results.join('\n') : 'No matching history found.';
  } catch (e) {
    return `Error in search_history: ${String(e)}`;
  }
}
```

**Changes from initial draft:**

- Added `JSON.parse(argsJson)` — `argsJson` is a raw string parameter, not a parsed object
- Added `try/catch` wrapper consistent with all other tool handlers
- Added empty query validation
- Included truncated message IDs in output so the LLM can follow up with `read_messages`
- Added "Use read_messages..." hint to enable natural workflow chaining

**Result caps:**

- Max 5 topic matches
- Max 5 message matches (content trimmed to 300 chars each)
- Results formatted as markdown for easy LLM consumption

**Performance consideration:** Each call loads all non-deleted messages from IndexedDB to build a Fuse index. For users with thousands of messages this may be slow. Mitigation options (implement if needed):

- Cache the Fuse index for the duration of a tool loop (invalidate on next user message)
- Limit scan to messages from the last 90 days
- Add a `limit` parameter the LLM can use to narrow scope
- Log timing to observe real-world impact before optimizing

**Rate limiting:** The existing tool loop limit (10 iterations) applies automatically. No additional rate limiting needed unless abuse is observed.

## Risk Mitigation

- **Privacy / scope:** Tool searches across ALL topics (unlike `read_messages`/`list_messages` which are current-topic only). Gated behind explicit user toggle (`historySearchEnabled`, default **false**). Toggle label explicitly states cross-conversation scope.
- **Token blowout:** Results capped at 5 per mode, content trimmed to 300 chars
- **Tool loops:** Natural limit of 10 iterations from `orchestrateLlmLoop`
- **Data freshness:** Data loaded from IndexedDB on each call (no stale cache)
- **Deleted data:** `isDeleted` filter prevents searching deleted topics/messages
- **Performance:** Full table scan + Fuse index build on every call. Acceptable for typical usage but may degrade with very large histories (10k+ messages). Performance section above documents mitigations to apply if needed.
- **Toggle dependency:** `historySearchEnabled` only takes effect when `messageRetrievalEnabled` is also on. UI should grey out the toggle when the parent is disabled.
